# Lucky iOS Panel · 最后联调关注点

## 1. 登录态切换是否清干净
### 重点看
- 登录后各页是否自动刷新
- 退出登录后 Dashboard / Services / Activity 是否立即清空旧数据
- 再次登录其他账号/其他地址后，页面是否出现前一次会话残留

### 当前已做保护
- `DashboardViewModel.load()` 在 token 为空时会清空：
  - `summaries = []`
  - `activities = []`
- `ServicesViewModel.load()` 在 token 为空时会清空：
  - `services = []`
- 请求失败时也会主动清空旧数据，避免“看起来还有数据，实际上已经失效”

---

## 2. token 失效后的页面表现
### 重点看
- Lucky 后台 token 过期后，是否能明确提示错误
- 页面是否会继续显示旧数据造成误判
- 是否需要在后续补自动跳转登录页

### 当前状态
- 现在优先保证：**失败就清空旧数据并显示错误**
- 还没有做全局“401/失效后强制跳登录”的最终 UX 收口

---

## 3. Services 页状态语义
### 重点看
- 当前：
  - `running` -> `主模块`
  - `available` -> `扩展模块`
- 这属于前端展示映射，不一定完全等于 Lucky 后端真实运行态

### 风险
如果 Lucky 后端后续返回更细状态，当前映射可能需要重做。

---

## 4. Dashboard 指标值是否稳定
### 重点看
- CPU、内存、连接数、模块数等是否持续有值
- 长文本/大数值是否会挤压 UI
- 某些字段缺失时是否还能优雅展示

### 当前状态
- 已加空态占位
- 已加 `minimumScaleFactor(0.85)` 缓解长值挤压
- 仍建议在 Xcode 真机/模拟器里跑一轮极端值观察

---

## 5. Activity 日志可读性与性能
### 重点看
- 日志较多时滚动是否流畅
- 日志正文是否可复制
- 时间戳/标题/正文分层是否足够清晰

### 当前状态
- `ActivityView` 已支持正文 `.textSelection(.enabled)`
- 当前展示适合联调与排障
- 如后续日志量继续加大，可再考虑分页或懒加载优化

---

## 6. LoginView 交互边界
### 重点看
- 登录中按钮是否重复点击
- 错误提示是否足够明确
- 登录成功后是否立刻切主界面

### 当前状态
- 登录按钮在 `isLoading` 时禁用
- 错误信息会直接展示
- 依赖 `SessionStore.isLoggedIn` 切换主界面

---

## 7. 本轮建议的最终验收顺序
1. 正常登录
2. 看 Dashboard 是否有真实数据
3. 看 Services 是否正确分组
4. 看 Activity 是否有真实日志
5. 退出登录，确认页面是否清空
6. 改错地址/密码，确认错误态
7. 再次登录，确认状态是否重新正确加载

---

## 结论
当前 Lucky iOS 面板已经进入“最后联调收口”阶段。
剩余重点不再是大功能开发，而是：
- 登录态边界验证
- 旧数据残留验证
- 展示语义与真实接口的一致性验证
